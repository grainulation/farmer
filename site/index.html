What is farmer?

When you run an AI coding agent like Claude Code, it sometimes needs your permission before executing commands or editing files. Normally you have to be sitting at your terminal. farmer removes that constraint.

It runs a local server that intercepts tool calls via hooks, streams permission requests to a web dashboard over WebSocket, and waits for your decision. You can approve from your phone, a different tab, or through a Cloudflare tunnel from anywhere in the world. Multi-session, trust-tiered, fully audited.

The adapter pattern means farmer is not locked to Claude Code. Any agent that can POST a JSON permission request to an HTTP endpoint can use it.

Launch farmer from your project directory. It prints a token-authenticated URL you can open on any device.

Hook protocol

Four hooks. Full lifecycle coverage.

farmer intercepts every stage of the agent tool-call lifecycle. Each hook receives a JSON payload via stdin and returns a JSON decision.

PreToolUse

Fires before a tool executes. This is where permission decisions happen -- approve, deny, or modify the request. The agent blocks until you respond.

Agent -> Farmer -> Decision

PostToolUse

Fires after a tool completes. Carries the result back so the dashboard can show what happened. Used for audit logging and result display.

Agent -> Farmer -> Logged

Notification

Non-blocking messages from the agent. Status updates, progress indicators, errors. Streams to the dashboard without requiring a response.

Agent -> Farmer -> Dashboard

Lifecycle

Session start and stop events. Tracks when agents connect and disconnect. Enables multi-session management and trust inheritance.

Agent -> Farmer -> Session
Architecture

Agent-agnostic by design

The adapter pattern separates the protocol layer from agent-specific details. Swap adapters to support any AI coding agent.

  AI Agent                  Farmer Server                Dashboard
  (Claude Code,              (Node.js, port 9090)          (Browser/Phone)
   any agent)
      |                           |                           |
      |  POST /hooks/permission  |                           |
      |-------------------------->|                           |
      |  {tool, input, session}  |                           |
      |                           |                           |
      |                           |  WebSocket push           |
      |                           |-------------------------->|
      |                           |  {card: pending}          |
      |                           |                           |
      |                           |       User taps Approve  |
      |                           |<--------------------------|
      |                           |  {decision: approve}      |
      |                           |                           |
      |  200 {decision: approve} |                           |
      |<--------------------------|                           |
      |                           |                           |
      |  POST /hooks/result     |  WebSocket push           |
      |-------------------------->|-------------------------->|
      |  {tool, output}          |  {card: completed}        |
      |                           |                           |
  ----|-------- SSE fallback --------|---
      |  GET /events (SSE)       |                           |
      |                           |  EventSource stream       |
      |                           |-------------------------->|
      |                           |  (for clients without     |
      |                           |   WebSocket support)       |

WebSocket is the primary transport for the dashboard. SSE provides a zero-dependency fallback. The hook endpoints use plain HTTP POST -- any agent that can curl can integrate.

Trust levels

Control how much your agent can do

Set per-session trust. New sessions inherit the trust level of their parent. Change it any time from the dashboard.

Paranoid

Every call reviewed

Nothing executes without explicit approval. Read, Write, Bash -- everything requires a tap. Best for sensitive repos or unfamiliar agents.

Standard

Read auto-approved

Read operations (file reads, glob, grep) are auto-approved. Write and execute operations require your approval. The default for most workflows.

Autonomous

Agent runs free

Everything is auto-approved. The dashboard still streams cards for monitoring, but nothing blocks. Use when you trust the agent fully.

Key features

Control without context-switching

Zero external dependencies beyond the ws library. No framework, no build step.

Token Authentication

Every session gets a unique token, persisted to disk so server restarts don't disrupt your workflow. No token, no access.

WebSocket + SSE

WebSocket for real-time bidirectional communication. SSE as a zero-dependency fallback. Permission cards stream the instant they are requested.

Multi-Session

Run multiple AI agents at once. Each gets its own session with independent trust levels. New sessions inherit trust from their parent.

Adapter Pattern

Agent-specific logic is isolated in adapters. Ship with a Claude Code adapter, swap in your own for other agents. The core protocol stays the same.

Audit Logging

Every approve, deny, and auto-decision is logged to a JSONL file with timestamps, session IDs, and tool details. Full accountability. harvest can analyze these logs for usage patterns.

Mobile-First

The dashboard is designed for phone screens first. Large tap targets, responsive layout, no zoom issues. Add to your home screen for native-like experience.

Cloudflare Tunnel

Built-in support for cloudflared tunnels. Expose your local server to the internet securely. Approve agent actions from anywhere in the world.

Sprint Detection

Watches claims.json and compilation.json for changes. The dashboard shows which wheat sprint your agent is working in. barn provides the detection logic.

Feedback Queue

Send feedback to your agent without switching to the terminal. The poll/ack pattern ensures messages are delivered even if the agent is mid-task.

Frequently asked questions

Does this only work with Claude Code?

No. farmer ships with a Claude Code adapter, but the architecture is agent-agnostic. Any tool that can POST a JSON permission request to an HTTP endpoint can use farmer. Write an adapter for your agent and plug it in.

How do I access it from my phone?

Three options: (1) If your phone is on the same network, bind to 0.0.0.0 and use your machine's local IP. (2) Use a Cloudflare tunnel -- farmer has built-in support. (3) Use ngrok, tailscale, or any tunnel of your choice. The token in the URL handles authentication.

What happens if I don't respond to a permission request?

The agent blocks. It waits until you approve or deny. There is no timeout by default -- your agent will not proceed without your decision. You can configure auto-approve rules per tool type via trust levels.

Does this work without wheat?

Yes. farmer works with any Claude Code session, not just wheat sprints. The sprint detection features are a bonus -- the core permission management works independently.
